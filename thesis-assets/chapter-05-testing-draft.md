# 第五章 系统测试（草稿）

## 5.1 测试目的与测试思路

系统测试的目的在于验证本系统在既定业务场景下的功能完整性、交互稳定性与基本可用性，并为后续论文结论与系统评价提供依据。由于本系统核心能力依赖外部大语言模型服务，模型生成内容本身具有一定开放性与非确定性，因此测试过程不能完全采用传统意义上的精确断言方式，而需要结合黑盒测试、边界测试与结果合理性检查，对系统的业务流程进行综合验证。

结合系统实现内容可知，本系统的关键功能链路包括用户身份认证、简历上传与解析、岗位模板驱动的会话创建、多轮流式面试交互以及总结性评估报告生成。其中，普通 REST 请求主要验证数据提交、权限校验和数据库写入是否正确，流式对话部分则重点关注连接建立、首字节响应、内容连续推送以及异常恢复机制是否稳定。因此，本章测试思路以“功能正确性优先，性能体验作为补充”展开，即先验证核心业务是否闭环，再通过首字节响应时间、解析耗时和流式展示效果评估系统在真实使用过程中的可接受性。

## 5.2 测试环境说明

本系统的测试环境以本地开发部署为主，主要由浏览器端、后端服务、数据库服务和外部模型调用环境构成。前端页面在浏览器中运行，后端基于 Spring Boot 启动服务，数据库采用 MySQL 8.0 存储业务数据，模型能力通过 DeepSeek API 获取。当前草稿阶段尚未嵌入你的真实环境参数，因此本节先采用系统实际运行所对应的通用结构描述；待后续补入真实测试环境后，可将其替换为最终论文版本中的正式配置说明。

建议最终在本节明确给出以下信息：操作系统版本、JDK 版本、MySQL 版本、浏览器版本以及网络环境说明。若系统测试主要在 Windows 环境下完成，则应按你当前机器的真实配置补齐具体版本号。对于依赖外部模型服务的流式接口测试，还应简要说明测试时是否处于校园网、家庭宽带或移动热点环境，以便解释模型响应时延的波动范围。

## 5.3 功能测试

功能测试重点验证系统的五类核心业务链路是否能够按预期完成。首先，在简历上传与解析场景下，应验证系统是否能够正确接收合法 PDF 文件、完成文本提取、调用外部模型并在页面展示结构化结果；同时，还需验证对非 PDF 文件、过大文件或解析失败场景的错误提示是否有效。其次，在面试初始化场景下，应重点检查用户是否能够基于已上传简历与岗位模板成功创建面试会话，系统是否能够写入首条 `system` 消息，并在历史会话列表中正确保留对应记录。

对于流式对话场景，功能测试不仅要验证用户输入能否成功触发模型回复，更要关注前端是否能以渐进式方式展示返回内容、按钮状态是否与连接状态联动、异常情况下是否能够提示“连接已断开，请重试”等信息。由于该模块是系统最核心的交互功能，因此测试时还应重点关注断流、接口异常或空内容输入时的边界表现，避免出现页面长时间卡死、发送按钮不可恢复或数据库消息写入不完整等问题。最后，在面试结束与报告生成场景中，需要验证系统是否能根据会话历史生成总结性报告，并正确更新会话状态，使用户能够在报告页及历史记录页查看完整结果。

根据当前系统实现，功能测试用例可围绕以下典型场景展开：正常 PDF 简历上传与解析、非 PDF 文件上传拦截、岗位选择后会话初始化、流式对话正常接收、结束面试后生成评估报告，以及空内容或恶意输入的拦截与提示。除此之外，出于安全性与完整性考虑，还应补充未登录访问受保护接口、跨用户访问他人简历或会话、以及 SSE 中断后前端状态恢复等测试结果。这类测试虽然不一定直接体现在页面主流程中，但在答辩问答和系统可靠性评估中具有较高重要性。

## 5.4 性能测试

本系统的性能测试不以大规模压测为目标，而是聚焦于真实使用场景下最关键的体验指标。对于模拟面试系统而言，用户对性能的直接感知主要集中在两个方面：一是提交回答后多久能够看到模型开始返回内容，即流式接口首字节响应时间；二是上传简历后系统需要多久才能完成文本提取与结构化解析。因此，本章性能测试建议重点记录 SSE 接口的 TTFB 和 PDF 解析耗时，并结合浏览器 DevTools 网络面板或接口测试工具进行说明。

在 SSE 接口测试中，可通过浏览器 DevTools 的 Network 面板记录流式请求从发送到收到首个响应字节的时间，并对同一场景进行若干次重复测试，计算一个更具代表性的平均值或区间范围。由于响应时间不仅受到本地运行环境影响，还会受到外部大语言模型服务状态、网络条件和会话上下文长度影响，因此论文中应如实说明样本量、测量条件和当时的网络环境，而不宜将少量样本测试结果包装为严格的高并发压测结论。

对于 PDF 解析耗时测试，可在简历上传场景下记录后端从接收文件到返回结构化结果之间的大致耗时，或进一步拆分为“文本提取耗时”和“模型结构化分析耗时”两个环节。若后续测试结果显示 PDFBox 文本提取耗时较短而模型调用耗时占主要比例，则可在结论部分指出本系统的主要性能瓶颈来自外部模型服务而非本地处理链路。这种测试分析方式更符合本系统的真实运行特点，也更适合毕业设计场景下的客观表述。

## 5.5 测试结论

从当前系统实现和测试设计可见，本系统能够围绕用户登录、简历解析、会话创建、流式对话与评估报告生成构建完整的业务闭环，具备较为明确的功能测试对象和可操作的验证路径。若后续真实测试结果表明主要功能均可稳定通过，则可以说明系统在核心场景下已具备基本可用性，能够满足模拟面试与简历诊断的设计目标。

同时，也应客观指出本系统的性能与稳定性在一定程度上受制于第三方大语言模型服务。例如，流式接口的首字节响应时间可能会受到网络环境、模型服务负载和上下文长度变化影响而产生波动；简历解析和报告生成的最终耗时也并非完全由本地系统决定。因此，在论文结论部分更适合将本系统评价为“在当前环境下能够稳定完成核心业务流程，并在可接受时间内提供交互反馈”，而不是将其表述为具备严格实时性或高并发能力的生产级平台。

总体而言，本章测试的核心价值在于证明系统是否能够按照设计预期稳定运行，并识别影响体验的关键瓶颈位置。待补入真实测试环境、真实通过情况和实际性能数据后，本章即可进一步收敛为终稿版本，并为第六章总结与展望提供客观依据。
