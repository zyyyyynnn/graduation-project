# 答辩材料包 2026-04-25

## 一、演示路线

1. 启动 Demo Twin：运行 `.\start-demo.bat`，说明 Demo 与真实模式端口、数据库和前端代理隔离。
2. 登录演示账号：使用 `demo / 123456` 进入系统。
3. 主工作台（报告预览）：展示当前简历、岗位、阶段化面试区、历史会话和报告预览。
4. 简历流程：上传 PDF 简历，说明后端 PDFBox 文本提取与结构化解析入口。
5. 面试流程：创建会话，按破冰、技术、深挖、收尾阶段推进问答。
6. 报告流程：生成 Markdown 评估报告，在主工作台（报告预览）与历史会话中查看结果。
7. 回放流程：进入 `/interview/replay/:sessionId`，展示阶段时间线与消息记录。
8. 分析流程：进入 `/analytics`，展示评分趋势、能力雷达和薄弱点统计。
9. 设置流程：简要展示 LLM 配置与用户设置，说明用户级模型配置和密钥保存边界。

## 二、系统亮点

| 亮点 | 答辩表述 |
| --- | --- |
| 完整业务闭环 | 系统覆盖登录、简历解析、岗位匹配、模拟面试、报告生成、回放和能力分析，不是单点页面展示。 |
| SSE 流式体验 | 面试回答采用 `fetch + ReadableStream` 接收流式内容，用户能看到渐进式返回效果。 |
| Demo Twin 机制 | Demo 模式使用独立端口、数据库和演示数据，便于答辩复现，同时不污染真实模式。 |
| 用户级 LLM 配置 | 支持用户维护 Provider、模型和 API Key，为后续扩展不同模型服务预留接口。 |
| 证据化测试 | 已完成 `TC-01` 到 `TC-09` 业务测试，并记录 SSE TTFB 与 PDF 上传/解析接口耗时。 |

## 三、关键难点

| 难点 | 解决思路 |
| --- | --- |
| 流式响应与消息落库一致性 | 后端在流式输出结束后统一收口 AI 回复，前端识别完成态并恢复按钮状态。 |
| 上下文长度控制 | 面试历史消息按窗口截断，避免请求上下文无限增长导致响应变慢或超出模型限制。 |
| Demo 与真实模式隔离 | 通过 `.env.demo`、Demo 数据库、端口和启动脚本校验保证演示链路可复现。 |
| 本地启动稳定性 | 启动脚本先检查配置、MySQL 和关键端口，再启动后端和前端，降低答辩现场失败风险。 |
| 页面一致性 | 通过统一页面壳层、卡片节奏、按钮层级和顶部菜单入口，减少页面跳转和视觉割裂。 |

## 五、追问问答口径

| 可能追问 | 回答口径 |
| --- | --- |
| 为什么使用 SSE，而不是 WebSocket？ | 本系统主要是服务端向前端持续推送模型文本，交互方向以单向流为主，SSE 实现更轻量，浏览器原生支持也更适合文本流式输出。 |
| Demo Twin 是否会影响真实业务？ | 不会。Demo 使用独立端口、`.env.demo`、Demo 数据库和 `/api/demo/reset` 重建数据，真实模式不自动插入演示用户。 |
| 如果外部模型服务不可用怎么办？ | 当前系统通过异常处理向前端返回错误提示，Demo Twin 可保证答辩演示不依赖公网模型；真实模式后续可扩展模型降级和重试策略。 |
| API Key 如何保存？ | 当前采用用户级配置，并通过后端加密后保存；论文中可说明这是毕业设计阶段的最小安全方案，生产环境仍需接入更完整的密钥管理。 |
| 测试数据是否代表真实公网性能？ | 不代表。当前 `TC-01` 到 `TC-09` 验证的是 Demo Twin 本机业务闭环，SSE TTFB 和 PDF 耗时用于证明链路可用；真实公网模型性能需要另测。 |
| 系统最大不足是什么？ | 主要不足是模型能力和响应时延依赖外部服务，且当前没有做大规模并发压测；后续可补模型降级、批量测试和部署环境监控。 |

## 六、演示风险与备用方案

| 风险 | 备用方案 |
| --- | --- |
| MySQL 未启动 | 先运行启动脚本准备阶段；若失败，根据脚本提示启动 MySQL 或修正本地配置。 |
| 浏览器缓存旧页面 | 强制刷新或重新运行 `start-demo.bat`，确认前端端口为 `5174`。 |
| 外部模型不可用 | 使用 Demo Twin 的脚本化演示数据完成全流程展示。 |
| 页面网络请求失败 | 先检查 `8081/api/health` 和 `5174/login`，再执行 `reset-demo.ps1` 重建演示数据。 |
