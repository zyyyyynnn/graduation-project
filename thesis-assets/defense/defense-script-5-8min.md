# 答辩正式讲稿（5-8 分钟版）

各位老师好，我的毕业设计题目是《基于 LLM 的模拟面试与简历诊断系统》。本系统面向求职准备场景，目标是把简历解析、岗位匹配、阶段化模拟面试、评估报告、历史回放和能力分析串成一条完整的训练闭环。

首先是选题背景。传统面试准备通常依赖人工辅导或简单题库，反馈成本较高，也难以持续记录每次训练中的表现变化。随着大语言模型的发展，系统可以根据简历、岗位和历史回答生成更接近真实面试的追问，并在结束后给出结构化反馈。因此，本课题希望实现一个可本地运行、可演示、可复盘的模拟面试系统。

系统总体采用前后端分离架构。前端使用 Vue 3 和 Element Plus，负责登录、简历管理、面试工作台、回放、数据看板和设置页面。后端使用 Spring Boot，提供认证、简历解析、面试会话、流式问答、报告生成和能力分析接口。数据层使用 MySQL 保存用户、简历、会话、消息、阶段、评分和薄弱点。外部能力主要包括 PDFBox 文本提取和 LLM Provider 调用。同时，为了保证答辩演示稳定，我设计了 Demo Twin 模式，它使用独立端口、独立数据库和可重置的演示数据，不依赖公网模型也不污染真实模式。

在业务流程上，用户登录后可以上传 PDF 简历，系统解析简历内容并与岗位模板结合创建面试会话。面试过程按照破冰、技术、深挖和收尾阶段推进。用户提交回答后，前端通过 `fetch + ReadableStream` 接收后端 SSE 流式内容，因此页面可以逐步展示模型回复，而不是等完整结果一次性返回。面试结束后，系统生成 Markdown 评估报告，并支持在历史会话中查看报告、回放消息和阶段时间线。数据看板会基于评分历史和薄弱点统计展示能力变化。

本系统的核心实现主要有三点。第一是流式面试链路。后端将模型返回内容按片段推送给前端，前端维护流式状态，并在收到完成标记后恢复按钮状态和刷新会话内容。第二是简历解析链路。系统使用 PDFBox 提取 PDF 文本，再将文本交给模型进行结构化解析，形成技能、项目和经历信息。第三是上下文控制。为了避免多轮面试历史无限增长，系统会对历史消息进行窗口截断，只保留与当前追问相关的上下文，从而控制请求长度和响应稳定性。

测试方面，我完成了后端单元测试、前端生产构建和 Demo Twin 业务测试。业务测试覆盖 `TC-01` 到 `TC-09`，包括合法 PDF 上传、非 PDF 拦截、面试初始化、流式对话、报告生成、空输入、未登录访问、越权访问和 SSE 异常恢复，结果全部通过。性能采集方面，在 Demo Twin 本机回环环境下，SSE 首个 chunk 到达时间为 59 ms，SSE 总耗时为 198 ms，PDF 上传与解析接口耗时为 36 ms。需要说明的是，这组数据用于证明本机业务闭环和前后端链路可用，不代表真实公网 LLM 服务的响应性能；公网模型性能后续需要在真实网络和真实 API Key 条件下单独采集。

开发过程中也遇到了一些真实问题。第一个是 Demo 登录时报 `ECONNREFUSED`。排查后发现是前端缺少 `.env.demo`，导致 Vite 代理默认指向真实后端端口，最终通过补齐 Demo 环境文件并在启动脚本准备阶段校验端口和代理目标解决。第二个是后端启动时 MySQL 连接失败。这个问题不是业务代码本身错误，而是数据库服务未就绪导致 Hikari 初始化失败。后续我把数据库连接、端口和关键密钥检查前置到启动脚本中，让错误更早暴露，避免等 Spring Boot 启动失败后才排查。

总结来说，本课题完成了一个可运行的模拟面试系统，重点体现了完整业务闭环、SSE 流式体验、Demo 与真实模式隔离、用户级 LLM 配置和证据化测试。当前不足是公网模型性能还没有在真实环境下采集，也没有做大规模并发压测。后续可以继续补充模型降级策略、批量测试、部署监控和更完整的密钥管理能力。

我的汇报结束，请各位老师批评指正。
