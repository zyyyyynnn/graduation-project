# 答辩准备与动态演练

## 目标

- 在你提供真实系统状态、截图、测试结果和个人开发经历后，由我生成答辩所需材料。
- 同时由协作模型负责构建题库、追问路径与模拟问答。

## 人工准备

1. 提供真实系统状态与真实素材：
   - 系统截图
   - 测试结果
   - 关键代码片段
   - 个人真实开发经历
2. 若有真实 Bug 排查记录，优先提供 `dev-notes.md` 内容。
3. 必须准备答辩容灾材料：
   - 2-3 分钟完整演示视频
   - 一套离线截图讲解顺序
   - 至少 2 个已验证可用的 API Key 或备用演示方案
4. 若人工侧已经准备了答辩 PPT 草稿，也可以一起提供给协作模型。
5. 若你希望答辩内容与论文完全对齐，建议同步整理答辩页映射表。

> 人工侧不需要自己先分别准备开场白、PPT、大题库或问答流程；把真实素材提供给协作模型，协作模型会统一生成。但演示视频、离线截图和备用方案必须提前准备好，否则现场风险无法转移。

## 可直接使用的提示词

### 8A：答辩静态材料生成

```markdown
我现在进入答辩准备阶段。以下是系统的真实信息、真实截图说明、真实测试结果和我的开发经历，请你直接帮我生成答辩所需静态材料。

已提供材料：
- 系统亮点：
[在此填写]
- 真实截图说明：
[在此填写]
- 真实测试结果：
[在此填写]
- 真实开发经历或真实 Bug 记录：
[在此填写]
- 答辩页映射表（如有）：
[在此填写]

请输出：
1. 60 秒开场白
2. PPT 大纲
3. 基础题库
4. 工作量与原创性防守话术
5. 演示失败时的离线讲解顺序与应对话术
```

### 8B：师生问答动态演练

```markdown
请扮演一位计算机专业的毕业设计评审老师，对我进行答辩提问。
系统信息：《基于大语言模型的沉浸式模拟面试与简历诊断系统》
（Vue 3 + Spring Boot 3 + SseEmitter + DeepSeek API）

规则：
1. 每次只问一个问题，等我回答后再继续
2. 根据我的回答质量决定后续：
   - 准确完整：给出简短肯定，换下一个方向
   - 有漏洞/含糊：立刻针对漏洞追问，最多深挖 2 层
   - 明显错误：指出错误，告知正确方向
3. 提问范围：SSE 实现机制、异步线程池、多轮上下文截断策略、系统异常处理、真实工作量、系统局限性
4. 问完 6 个问题后，给我一个总体评价，并指出最薄弱的 1-2 个知识点

请作为老师，向我提出第一个问题。
```

## 使用提示

- 建议先完成静态材料，再开始动态演练。
- 动态演练效果取决于你提供的真实开发经历和真实 Bug 记录，材料越真实，追问越贴近答辩现场。
- 若你希望我根据某位老师的偏好或某类高频问题定制题库，可以直接补充说明。
- 不要把答辩演示完全押在实时联网调用上；离线视频和截图讲解顺序应视为必备材料。
- 若你想把 PPT 每页和论文章节、截图、代码绑定起来，可先复制 [assets-template/defense-slide-map-template.md](./assets-template/defense-slide-map-template.md)。
